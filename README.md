# VendorFlow

**Русский** · [English](README.en.md)

AI-агент, который **по команде менеджера первым пишет подрядчикам в Telegram**, ведёт короткий деловой диалог, собирает цену, сроки и условия, фиксирует всё в Google Sheets и **останавливается на ручную проверку, когда нужно человеческое решение**. Пишет естественно — в рабочее время, с паузами, как живой человек.

Прототип для тестового задания на позицию AI-интегратора.

---

## Что делает (за 30 секунд)

1. Менеджер добавляет вендора в Google-таблицу: **логин, задача** и ставит галочку **«Запустить»**.
2. Планировщик сам, в рабочее время и с человеческими паузами, пишет подрядчику первым.
3. Агент ведёт диалог: задаёт по одному вопросу, **извлекает цену / срок / условия**, пишет результат в ту же строку таблицы.
4. Если всё собрано — помечает **«готово»**. Если нужен человек (просят созвон, оплату, договор, или цена выглядит неадекватно) — **останавливается, ставит «на проверку» и зовёт менеджера в Telegram-бот**, подрядчику при этом ничего не пишет.

Таблица — единственный вход и вся CRM: одна строка = один вендор.

| Telegram | Задача | Запустить | Статус | Цена | Срок | Условия |
|---|---|---|---|---|---|---|
| @vendor_a | ролик ~60с | ✅ | ждём ответа | 5000 | неделя | — |
| @vendor_b | нарезка shorts | ✅ | на проверку | — | — | — |

## Попробовать без всего (одна команда, без ключей и Telegram)

```bash
pip install -r requirements.txt
python demo.py
```

`demo.py` прогоняет **настоящий движок** на трёх подрядчиках, но с заглушками вместо Telegram / Google / LLM — видно, как агент пишет первым, собирает данные и в двух разных случаях останавливается на ручную проверку. Кредов не нужно.

## Как устроено (ключевые решения)

| Модуль | Файл | Задача |
|--------|------|--------|
| Оркестратор | `vendorflow/engine.py` | переходы статусов, вся логика решений |
| Планировщик | `vendorflow/scheduler.py` | рабочие часы, паузы, лимиты, kill switch — обычный код, не LLM |
| LLM-анализ | `vendorflow/llm_pipeline.py` → `analyze()` | извлечение данных + стоп-правила, temperature=0, только JSON |
| LLM-текст | `vendorflow/llm_pipeline.py` → `compose()` | сочиняет сообщение, вызывается только если анализ разрешил |
| Хранилище | `vendorflow/state_store.py` | Google Sheets, дедуп, идемпотентность |
| Отправка | `vendorflow/telegram_client.py` | Telethon (пишет первым), имитация набора |
| Бот менеджера | `vendorflow/manager_bot.py` | `/contact`, `/status`, `/stop`, уведомления |

**Почему два отдельных вызова LLM.** Если решать «останавливаться ли» и сочинять ответ в одном вызове, модель склонна продолжить диалог ради связного текста. Холодный анализатор (temperature=0), не занятый задачей «написать красиво», судит трезвее; текст сочиняется отдельно и только когда анализ разрешил.

**Почему планировщик — не LLM.** У модели нет надёжных часов и памяти между вызовами. «Можно ли и когда отправлять» решает код: рабочие часы Пн–Пт 10:00–18:00, паузы 1–5 мин, дневные лимиты. LLM только предлагает сообщение и статус.

**Провайдер LLM абстрактный.** По умолчанию `claude-code` (локальный CLI, без API-ключа); переключается на OpenAI / OpenRouter одной переменной в `.env`. Разные модели на разные задачи: анализ — быстрая, сочинение — качественнее.

## Безопасность

- `DRY_RUN=true` (по умолчанию) — подрядчикам ничего не уходит, всё в логи.
- `KILL_SWITCH=true` — мгновенно глушит любую отправку.
- `TEST_RUN=true` — для отладки обходит рабочие часы/паузы/лимиты (но **не** kill switch).
- Стоп-правила (созвон, оплата/реквизиты, договор, недовольство, аномалии, неуверенность модели) → статус `needs_human_review`, подрядчику ничего, зовётся менеджер.
- Дедуп: один подрядчик = одна строка по нормализованному telegram-ключу; первое сообщение идемпотентно.

## Полный запуск (с реальными Telegram и Google)

```bash
pip install -r requirements.txt
cp .env.example .env      # заполнить ключи (см. комментарии в файле)
python auth.py            # разовый логин Telethon-аккаунта (сохранит сессию)
pytest -q                 # прогнать тесты
python main.py            # live-режим (пока DRY_RUN=true — безопасно)
```

Нужно в `.env`: доступ к Google-таблице (`SHEET_ID` + сервисный аккаунт), `TG_API_ID`/`TG_API_HASH` (my.telegram.org), `MANAGER_BOT_TOKEN`/`MANAGER_CHAT_ID`. Ключ OpenAI не обязателен — по умолчанию работает через локальный `claude-code`.

## Статусы подрядчика

`в очереди → пишем → ждём ответа → (на проверку | готово)`, плюс `стоп` — ручное выключение менеджером.
